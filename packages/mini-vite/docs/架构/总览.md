## 核心分层

`mini-vite` 可以按职责拆成六层：

1. CLI 与启动入口
2. 开发服务器与中间件
3. 插件系统与插件容器
4. 模块转换链路
5. HMR 与模块图
6. 依赖预构建

## 对应源码入口

- `bin/mini-vite.js`：命令行入口
- `src/node/cli.ts`：CLI 参数解析与命令分发
- `src/node/server/index.ts`：服务启动、插件注册、中间件装配
- `src/node/server/middlewares/indexHtml.ts`：HTML 请求与 HMR 客户端注入
- `src/node/server/middlewares/transform.ts`：JS/CSS 请求拦截与转换入口
- `src/node/transformRequest.ts`：`resolveId -> load -> transform` 主流程
- `src/node/server/pluginContainer.ts`：插件容器
- `src/node/plugins/*.ts`：各类资源与能力插件
- `src/node/server/hmr.ts`：文件监听、模块图、消息广播
- `src/client/client.ts`：浏览器端 HMR 运行时
- `src/node/optimizer/index.ts`：依赖预构建

## 模块协作关系

服务启动后，大致会形成下面这条主链路：

1. CLI 调用开发服务器启动逻辑
2. 服务端创建插件数组与插件容器
3. 中间件按顺序接管 HTML、静态目录与模块请求
4. 模块请求进入 `transformRequest`，依次经过 `resolveId`、`load`、`transform`
5. `importAnalysis` 在转换阶段重写导入并维护模块图
6. `hmr` 服务监听文件变化，根据模块图向客户端广播更新
7. `optimizer` 在启动后扫描裸导入并输出预构建产物

## 读源码时可以重点抓的接口

- `Plugin` 接口：看清系统允许插件介入哪些阶段
- `createPluginContainer`：看清插件钩子是怎么被串起来的
- `transformRequest`：看清单个模块请求的完整闭环
- `createHMRServer`：看清文件变化如何映射到客户端更新
- `optimizeDeps`：看清裸导入为什么会被重写到 `.mini-vite/deps`
