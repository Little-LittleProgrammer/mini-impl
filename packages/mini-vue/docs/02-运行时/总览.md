## 1. 运行时分为哪几层

这一部分主要由两个包组成：

- `runtime-core`
  平台无关，负责 VNode、组件、渲染器、调度器
- `runtime-dom`
  浏览器平台适配，负责把抽象宿主操作映射到 DOM API

其中 `runtime-core` 是真正的核心，`runtime-dom` 只是把“如何操作节点”这件事补齐。

## 2. 入口链路

用户侧典型调用链可以概括成：

```text
createApp(App).mount('#app')
-> createVNode(rootComponent)
-> render(vnode, container)
-> patch(null, vnode, container)
-> 根据 vnode 类型进入元素 / 组件 / 文本 / Fragment 分支
-> 生成真实 DOM
```

这里真正的总调度函数是 `renderer.ts` 里的 `patch`。

## 3. VNode 是怎么设计的

`runtime-core/src/vnode.ts` 负责统一虚拟节点形态。

### 3.1 关键字段

一个 vnode 至少包含：

- `type`
- `props`
- `children`
- `key`
- `shapeFlag`

### 3.2 为什么要有 `shapeFlag`

`shapeFlag` 的作用，是把一些高频判断从运行时反复做类型推断，改成一次编码、多次按位判断。

例如：

- `ELEMENT`
- `STATEFUL_COMPONENT`
- `TEXT_CHILDREN`
- `ARRAY_CHILDREN`

这样 `patch`、`mountElement`、`patchChildren` 里都可以快速分发。

### 3.3 当前 children 支持范围

当前实现稳定覆盖的只有：

- 文本 children
- 数组 children

源码里对下面几类都还只是占位：

- 对象 children
- 函数 children
- 插槽 children

所以这份 mini-vue 还不能把“组件 children = slots”这套能力真正建立起来。

## 4. `h()` 的定位

`runtime-core/src/h.ts` 提供了一个轻量版 `h()`：

- 处理 `h(type, props)`
- 处理 `h(type, children)`
- 处理 `h(type, props, children)`
- 处理多个 children 参数

它的目标很明确：降低手写 render 函数的成本。

## 5. 渲染器 `createRenderer` 的核心思想

### 5.1 平台无关的 renderer

`createRenderer(options)` 会接收宿主配置，然后返回：

- `render`
- `createApp`

这和真实 Vue 的自定义渲染器设计是一致的：核心算法不绑定 DOM，只绑定一组 host APIs。

### 5.2 `patch` 是总入口

`patch(oldVNode, newVNode, container, anchor)` 会按节点类型分流到：

- `processText`
- `processCommentNode`
- `processFragment`
- `processElement`
- `processComponent`

## 6. 元素节点的挂载与更新

### 6.1 挂载

`mountElement` 的顺序很直接：

1. `hostCreateElement(type)` 创建元素
2. 根据 `shapeFlag` 挂载文本或数组子节点
3. 遍历 `props` 调用 `hostPatchProp`
4. `hostInsert` 插入容器

### 6.2 更新

`patchElement` 主要做两件事：

1. `patchChildren`
2. `patchProps`

### `patchProps` 已实现的点

- 遍历新 props，做新增或更新
- 遍历旧 props，删除新 props 中不存在的键

### `patchChildren` 当前真实状态

已具备的能力：

- 文本 children 更新
- 数组 children 之间 keyed diff
- 前后双端预处理
- 乱序 diff
- 最长递增子序列优化移动次数

未完成或明显简化的能力：

- 旧数组到新文本时，旧 children 的卸载是 `TODO`
- 旧数组到空 children 时，卸载是 `TODO`
- 旧文本到新数组时，挂载新 children 是 `TODO`
- `unmount` 逻辑过于简单，很多复杂节点类型不会正确清理

因此准确说法不是“它有完整 diff”，而是“它已经实现了 keyed diff 的主算法，但外围分支并未补齐”。

## 7. keyed diff 的原理

处理顺序是：

1. 从头部同步
2. 从尾部同步
3. 处理只新增
4. 处理只删除
5. 处理中间乱序段

乱序段里又会做三件事：

1. 建立 `key -> newIndex` 映射
2. 遍历旧节点，判断复用还是卸载
3. 根据 `newIndexToOldIndexMap` 计算最长递增子序列，只移动必要节点

这部分是运行时里最接近真实 Vue 思路的实现。

## 8. 组件系统的实现方式

### 8.1 组件实例很轻

`createComponentInstance` 生成的实例只包含最基础字段：

- `uid`
- `vnode`
- `type`
- `subTree`
- `effect`
- `update`
- `render`
- `isMounted`
- 生命周期相关字段

没有完整 Vue 那些复杂的实例数据层，例如：

- `props`
- `attrs`
- `slots`
- `setupState`
- `ctx`
- `proxy`
- `provides`
- `emit`

### 8.2 `setupComponent` 很薄

当前组件初始化主要做三件事：

1. 执行 `setup`
2. 确定 `instance.render`
3. 应用 Options API 风格的 `data / beforeCreate / created / beforeMount / mounted`

需要特别注意的点：

- `setup()` 当前没有拿到 `props`
- 也没有 `setupContext`
- 如果 `setup()` 返回函数，该函数会被当成 `render`
- 如果组件没有 `render`，但有 `template`，并且运行时注册过编译器，就会把 `template` 编译成 `render`

### 8.3 组件更新是如何发生的

组件在挂载时会创建一个 `ReactiveEffect(componentUpdateFn, scheduler)`。

它的意义是：

1. 首次 `effect.run()` 时执行挂载
2. 渲染过程中访问到的响应式数据完成依赖收集
3. 后续数据变化时，`trigger` 找到这个 effect
4. scheduler 把 `update` 放进微任务队列
5. 下一轮微任务里重新执行 render，再 patch 新旧子树

### 8.4 当前组件更新的边界

虽然组件 effect 已经建立，但组件能力还远不完整：

- `processComponent` 只处理初次挂载，没有完整的组件更新分派逻辑
- `instance.next` 在更新分支里被读取，但外围没有完整的新 vnode 进入流程
- 没有 props 更新、slots 更新、attrs fallthrough
- 没有 `shouldUpdateComponent`
- 没有组件级卸载和副作用清理

所以现在更准确的描述是：

- “组件内部响应式状态变化时，组件可以自更新”
- 但“父组件驱动子组件 props 更新”这一整套还没有真正成形

## 9. 生命周期实现

当前只覆盖了非常少的一部分：

- `beforeCreate`
- `created`
- `beforeMount`
- `mounted`

并且实现方式很直接：

- 选项式生命周期通过 `applyOptions`
- 组合式生命周期通过 `injectHook`

没有以下能力：

- `beforeUpdate / updated`
- `beforeUnmount / unmounted`
- `errorCaptured`
- `activated / deactivated`
- `onRenderTracked / onRenderTriggered`

## 10. 调度器为什么重要

`scheduler.ts` 当前只维护了一个 `pendingPreFlushCbs` 队列，并借助 `Promise.resolve().then(...)` 放入微任务。

它的效果是：

- 同一轮同步修改里，多次触发的更新会被合并
- 避免每次 `set` 都立刻同步重渲染

但它仍然非常薄：

- 只有 pre-flush，没有 job queue / post-flush
- 没有优先级
- 没有递归更新保护
- 没有组件级排序
- 没有 `nextTick`

## 11. `runtime-dom` 做了什么

`runtime-dom` 干的事非常聚焦：提供 DOM 平台的宿主操作。

### 11.1 `nodeOps`

把抽象操作映射到浏览器 API：

- `insert -> parent.insertBefore`
- `createElement -> document.createElement`
- `remove -> removeChild`
- `createText -> document.createTextNode`
- `createComment -> document.createComment`

### 11.2 `patchProp`

按属性种类细分：

- `class -> patchClass`
- `style -> patchStyle`
- `onXxx -> patchEvent`
- DOM property -> `patchDOMProp`
- 其他 -> `patchAttr`

### 11.3 当前 DOM 层的简化点

- 事件系统只有最基础 invoker 缓存
- `patchStyle` 只处理对象 style，字符串 style 和删除场景不完整
- `patchDOMProp` 只是 try/catch 直接赋值
- 缺少很多边界兼容

## 12. 运行时整体评价

如果只看运行时，这个 mini-vue 已经把最核心的教学闭环跑通了：

1. 用 vnode 描述界面
2. 用 renderer 把 vnode 变成 DOM
3. 用 reactive/effect 驱动组件重渲染
4. 用 scheduler 做最基础的异步批处理

但它离完整 Vue 的差距也非常明确：

- 组件模型是极简版
- 渲染器外围分支没补齐
- 卸载与更新流程偏粗糙
- 缺少大量生产级边界处理
