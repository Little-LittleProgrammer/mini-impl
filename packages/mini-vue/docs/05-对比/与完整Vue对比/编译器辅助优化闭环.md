# 编译器辅助优化闭环

## 1. 为什么完整 Vue 要让编译器和运行时协作

如果运行时每次更新都把整棵子树当成“未知结构”去比较，成本会很高。完整 Vue 的做法是让编译器在构建阶段先分析模板，把优化信息直接编码进生成的渲染函数。

## 2. 完整 Vue 的实现原理

完整 Vue 的编译器会识别：

- 哪些节点是静态的
- 哪些 props 是动态的
- 哪些子节点需要频繁更新
- 哪些事件处理器或插槽函数可以缓存

### 1. `patchFlag`（补丁标志）

`patchFlag` 是 Vue 3 中用于优化 diff 算法的核心机制。它是一个整数值，在编译阶段生成，用于标记节点中哪些部分是动态的（可能会改变）。

*   **作用**：在传统的虚拟 DOM diff 算法中，框架需要遍历并比较节点的所有属性（class, style, props 等）。而在 Vue 3 中，编译器会在生成代码时分析模板，如果发现某个节点的文本内容是动态的，就会给该节点打上“文本动态”的标志；如果 class 是动态的，就打上“class 动态”的标志。
*   **优化效果**：运行时在进行更新时，无需深入检查节点的所有细节，只需根据 `patchFlag` 有针对性地比对发生变化的部分。这极大地减少了 DOM 操作的工作量。

### 2. `dynamicProps`（动态属性）

`dynamicProps` 是一个字符串数组，包含了当前节点上所有动态属性的名称。

*   **作用**：当 `patchFlag` 表明该节点有动态属性时，运行时需要知道具体是哪些属性发生了变化。`dynamicProps` 数组列出了这些属性的 key（例如 `['class', 'id']`）。
*   **优化效果**：配合 `patchFlag`，Vue 在更新节点时不需要遍历节点上的所有属性，只需遍历 `dynamicProps` 数组中指定的属性进行对比和更新，实现了属性层面的“靶向更新”。

### 3. Block tree（块树）

Block tree 是 Vue 3 引入的一种新的虚拟 DOM 节点结构组织方式。

*   **定义**：一个“Block”是一个特殊的 VNode，它能够追踪其所有动态后代节点。在 Vue 3 中，所有的模板块（如组件模板根节点、带有指令的元素等）都被视为 Block。
*   **机制**：传统的虚拟 DOM 树在更新时需要递归遍历整棵树。而在 Block tree 机制下，一个 Block 节点会收集其内部所有的动态子节点，并将它们存储在 `dynamicChildren` 数组中。
*   **优化效果**：当响应式数据发生变化触发更新时，Vue 只需要从 Block 节点开始，遍历其 `dynamicChildren` 数组即可，完全跳过了对静态节点的遍历。这实际上是将树形结构的 diff 过程“扁平化”处理，大幅提升了更新效率。

### 4. `dynamicChildren`（动态子代）

`dynamicChildren` 是 Block 节点上的一个数组属性，用于存储该 Block 范围内所有的动态后代节点。

*   **作用**：它是 Block tree 机制的具体实现载体。当 Vue 编译器生成 VNode 时，如果一个节点是动态的，它就会被添加到父级 Block 的 `dynamicChildren` 数组中。
*   **优化效果**：这个数组是一个扁平化的列表。在进行 diff 比较时，Vue 直接遍历这个数组，而不需要递归遍历整棵树。这意味着无论模板层级有多深，只要确定了一个 Block，就能直接定位到所有需要更新的节点，实现了时间复杂度从 O(n) 到近乎 O(1)（针对特定动态节点查找）的优化。

## 3. 为什么这是一条闭环而不是单边优化

只有编译器输出了这些标记，运行时才知道如何跳过工作；只有运行时愿意识别并消费这些标记，编译器分析结果才有价值。所以完整 Vue 的优化不是“编译器更聪明”这么简单，而是编译器和 renderer 共同约定了一套协议。

## 4. 理解重点

完整 Vue 的高性能 VDOM，不是靠 renderer 单独变复杂，而是靠“模板静态分析 + 运行时定向执行”形成闭环。
