# 编译器辅助优化闭环

## 1. 为什么完整 Vue 要让编译器和运行时协作

如果运行时每次更新都把整棵子树当成“未知结构”去比较，成本会很高。完整 Vue 的做法是让编译器在构建阶段先分析模板，把优化信息直接编码进生成的渲染函数。

## 2. 完整 Vue 的实现原理

完整 Vue 的编译器会识别：

- 哪些节点是静态的
- 哪些 props 是动态的
- 哪些子节点需要频繁更新
- 哪些事件处理器或插槽函数可以缓存

这些信息会变成运行时可读的结构，例如：

- `patchFlag`
- `dynamicProps`
- block tree
- `dynamicChildren`

运行时拿到这些信息后，可以明显缩小更新范围。

`patchFlag` 用来描述一个 vnode 上真正可能变化的部分，比如：

- 文本变化
- class 变化
- style 变化
- 指定 props 变化

于是 renderer 在 patch 时可以直接走定向更新逻辑，而不必重新全量比对。

block tree 的核心思想是：把“有动态内容的节点”单独收集起来。这样父节点更新时，不需要深度遍历整个静态子树，只需要遍历当前 block 记录的动态子节点集合。

静态提升则会把完全不变的 vnode 或 props 提升到渲染函数外，避免每次渲染重复创建对象。

## 3. 为什么这是一条闭环而不是单边优化

只有编译器输出了这些标记，运行时才知道如何跳过工作；只有运行时愿意识别并消费这些标记，编译器分析结果才有价值。所以完整 Vue 的优化不是“编译器更聪明”这么简单，而是编译器和 renderer 共同约定了一套协议。

## 4. 理解重点

完整 Vue 的高性能 VDOM，不是靠 renderer 单独变复杂，而是靠“模板静态分析 + 运行时定向执行”形成闭环。
