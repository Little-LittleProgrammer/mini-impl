# 内建组件与高级能力

## 1. 这些能力为什么不能当普通组件处理

`Teleport`、`KeepAlive`、`Suspense`、`Transition`、`TransitionGroup` 看起来像组件，但它们本质上都在改写渲染器行为：

- 改节点挂载位置
- 改组件缓存策略
- 改异步依赖协调方式
- 改 DOM 进入和离开时机

因此完整 Vue 会把它们当作“带特殊分支的内建组件”处理。

## 2. 完整 Vue 的实现原理

`Teleport` 的重点不是多渲染一次，而是把逻辑父子关系和物理 DOM 容器拆开。渲染器在处理它时，会：

- 在原位置保留锚点
- 把子节点挂到目标容器
- 在目标变化时执行搬移
- 卸载时同时清理目标容器中的内容

`KeepAlive` 的核心是缓存组件子树，而不是重复创建。它会：

- 按组件类型或 key 建缓存表
- 把组件实例和渲染子树一起保留
- 失活时把真实 DOM 挪走但不销毁实例
- 再次激活时直接复用旧实例和旧子树

`Suspense` 的目标是协调异步依赖。它会同时维护：

- 主内容分支
- fallback 分支
- 异步依赖计数

当异步 setup 或异步组件尚未完成时，先展示 fallback；等依赖都 resolve 后，再把主内容切回来。

`Transition` / `TransitionGroup` 则深度介入 DOM patch。它们会在插入、移除、移动节点前后插入 class 切换和时序控制，并监听过渡结束，再决定何时真正移除 DOM。

## 3. 为什么这类能力会把渲染器复杂度迅速拉高

这些内建能力的共同点是：它们不只是“多一层组件抽象”，而是要改变 renderer 对挂载、移动、缓存、卸载的默认规则。所以完整 Vue 需要在 vnode 类型判断、shape flag、patch 流程、宿主操作之间预留很多专门分支。

## 4. 理解重点

这类高级能力之所以难，不在于 API 名字多，而在于它们都要求“组件层语义”直接影响“渲染器底层行为”。
