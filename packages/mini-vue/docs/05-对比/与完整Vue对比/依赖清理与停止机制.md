# 依赖清理与停止机制

## 1. 为什么完整 Vue 必须重视 cleanup

如果 effect 每次执行都只新增依赖、不移除旧依赖，就会产生两个问题：

- 条件分支切换后，effect 还订阅着已经不再访问的属性
- 停止 effect 后，依赖集合里还残留失效订阅

这既会带来错误触发，也会带来内存滞留。

## 2. 完整 Vue 的实现原理

完整 Vue 的 `ReactiveEffect` 一般会反向记录“自己当前被哪些 dep 持有”。于是每次 effect 重新执行前，都能先做一轮 cleanup：

1. 遍历自己记录的 dep 列表
2. 把自己从每个 dep 集合里移除
3. 清空本轮旧依赖记录
4. 再重新执行函数，建立新一轮依赖关系

这样条件分支变化时，依赖关系会跟着当前执行路径刷新，而不是越积越多。

`stop()` 的实现也是建立在这套反向索引上。停止时会：

- 对当前 effect 做最后一次 cleanup
- 标记 effect 为 inactive
- 执行用户注册的 `onStop`

之后即使响应式数据再次触发到旧 dep，也不会再调度这个 effect。

完整 Vue 还会配合位标记、批量清理等手段降低 cleanup 成本，避免 effect 数量多时每次全量重扫造成明显性能损耗。

## 3. 为什么这部分是生产能力分水岭

没有 cleanup，很多演示案例仍然能跑；但在真实组件树和复杂条件分支下，依赖会越来越脏。完整 Vue 必须保证“依赖关系是当前执行结果的真实投影”，而不是历史残留。

## 4. 理解重点

响应式系统的稳定性，不只取决于能不能 `track / trigger`，还取决于能不能把过期依赖及时断开。
