# 集合类型支持

## 1. 为什么 `Map / Set` 不能直接复用对象代理逻辑

普通对象的响应式主要围绕“属性读取和设置”展开，但集合类型真正重要的操作是：

- `get`
- `set`
- `add`
- `delete`
- `clear`
- `forEach`
- 迭代器遍历
- `size`

它们并不直接走普通属性访问路径。

## 2. 完整 Vue 的实现原理

完整 Vue 对集合类型通常会使用专门的 instrumentation，而不是直接把原始方法暴露出去。核心做法是：

- 在读取 `size` 时收集一个与迭代相关的依赖
- 在调用 `get`、`has` 时追踪指定 key
- 在 `set`、`add`、`delete`、`clear` 时区分“新增”“修改”“删除”三类触发
- 在迭代相关方法里追踪专门的迭代 key

这样才能保证下面这些场景的更新都正确：

- `map.get(key)` 只在这个 key 变化时重跑
- `for...of map` 会在键值对集合变化时重跑
- 读取 `set.size` 会在元素增删时重跑

完整 Vue 还要处理集合元素本身的代理转换。例如从 `reactive(map)` 里读出的对象值，仍然要根据场景返回响应式代理或只读代理。

## 3. 为什么这会增加很多边界

集合类型最大的难点是“依赖维度比对象属性更多”：

- 某个具体 key
- 整体迭代结构
- `size`

如果这些维度混在一起，触发就会过多；如果区分不清，又会漏更新。

## 4. 理解重点

完整 Vue 对集合类型的支持，本质上是在为不同集合操作建立更细粒度的依赖语义。
