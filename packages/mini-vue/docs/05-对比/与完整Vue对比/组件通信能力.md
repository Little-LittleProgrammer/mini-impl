# 组件通信能力

## 1. 完整 Vue 为什么把通信拆成多条通道

组件之间并不是只有一种数据流。完整 Vue 至少把通信拆成：

- `props`：父传子、声明式输入
- `emits`：子传父、事件输出
- `slots`：父向子传结构和渲染逻辑
- `provide / inject`：跨层级依赖传递
- `attrs fallthrough`：未声明输入的自动透传
- `expose`：对子组件公开有限实例能力

这么拆分是为了让“数据”“事件”“结构”“依赖”各自走最合适的通道。

## 2. 完整 Vue 的实现原理

`props` 的核心实现不是简单复制对象，而是：

- 先根据组件声明做标准化
- 区分 `props` 和 `attrs`
- 给 `props` 保持响应式更新入口
- 在父组件更新时按声明规则重新比对和刷新

`emits` 机制的重点是把组件对外事件也纳入声明系统。内部 `emit` 会：

- 规范化事件名
- 校验事件是否已声明
- 查找父组件传入的事件处理器
- 按约定处理修饰后的监听名

`slots` 在完整 Vue 里本质上是函数对象，而不是一段静态子节点。父组件会把插槽编译成函数，子组件渲染时再调用这些函数拿到子树。这样插槽才能拿到子组件传回的 slot props，也才能保持精确更新。

`provide / inject` 的关键实现是原型链继承。组件实例初始化时，其 `provides` 默认指向父组件的 `provides`。只有当前组件第一次调用 `provide` 时，才创建一个以父级 `provides` 为原型的新对象。这样：

- 查询是天然按层向上回退
- 没有写入时不会额外复制
- 作用域边界也很清晰

`attrs fallthrough` 则发生在渲染阶段。组件实例会保留未声明属性，渲染单根元素时自动把这些属性合并到根节点上；如果是多根节点，则需要显式处理或给出提示。

`expose` 的本质是建立一份受控公开对象。父组件通过模板 `ref` 拿到的不是全部内部实例，而是经过 `expose` 过滤后的公共接口。

## 3. 为什么 mini-vue 通常不会一次做全

教学实现里，组件先跑通父子渲染关系就已经足够。可一旦把通信做完整，就必须同时补齐：

- 组件声明标准化
- 实例代理
- 更新流程
- 插槽函数化
- 依赖注入上下文

这些能力是连在一起的，不能只单独补一层表面 API。

## 4. 理解重点

完整 Vue 的组件通信系统，本质上是在用不同语义的通道，分别处理输入、输出、结构复用和跨层依赖。
