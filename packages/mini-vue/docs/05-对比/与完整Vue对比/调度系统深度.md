# 调度系统深度

## 1. 调度器在完整 Vue 里负责什么

完整 Vue 的 scheduler 不是“把更新丢进一个微任务”这么简单。它要解决：

- 多次同步状态变更的批量合并
- 父子组件更新顺序稳定
- 用户 watcher 和组件更新的前后时机
- flush 过程中再次产生任务的重入处理
- `nextTick`

## 2. 完整 Vue 的实现原理

完整 Vue 通常会把待执行任务拆成几类队列：

- 组件更新 job 队列
- `preFlush` 回调队列
- `postFlush` 回调队列

`queueJob()` 不会立刻执行，而是先去重，再按 id 或组件创建顺序插入主队列。这样可以保证：

- 同一个组件一轮内最多更新一次
- 父组件通常先于子组件更新
- 已被卸载的组件 job 能被跳过

`preFlush` 主要给某些 watcher 或渲染前副作用使用，`postFlush` 则适合 DOM 更新后再执行的逻辑。真正 flush 时，完整 Vue 会严格控制顺序：

1. 先跑 `preFlush` 回调
2. 再跑组件更新 job
3. 最后跑 `postFlush` 回调

同时 scheduler 还要处理递归保护。如果某个 job 在一次 flush 里不断重新把自己塞回队列，就要有次数限制和警告，避免死循环。

`nextTick` 的实现则依赖当前 flush 使用的 Promise 微任务。当这轮队列清空后，再执行用户传入回调，于是用户能拿到“DOM 已经按本轮更新完成”的时机。

## 3. 为什么这部分会直接影响框架体验

如果没有完整调度系统，就会出现很多边界问题：

- 同一状态改三次，组件渲染三次
- 父子组件更新顺序错乱
- watcher 比 DOM 更新更晚或更早
- 异步回调里拿不到稳定 DOM

所以 scheduler 本质上是在维护整套响应式更新的执行秩序。

## 4. 理解重点

完整 Vue 的调度器管理的不是“任务列表”，而是“组件更新与用户副作用的全局时序”。
