# 应用实例能力

## 1. 完整 Vue 要解决什么问题

完整 Vue 里的 `app` 不是简单的“把根组件挂上去”，而是整个应用级上下文的入口。它要负责：

- 管理全局配置
- 注册插件、组件、指令
- 建立依赖注入根上下文
- 约束一个应用只挂载一次
- 为运行时和开发调试提供统一入口

## 2. 完整 Vue 的实现原理

完整 Vue 的 `createApp` 会先创建一个 `AppContext`。这个上下文通常集中保存：

- `config`
- 全局 `mixins`
- 全局 `components`
- 全局 `directives`
- 应用级 `provides`
- 插件去重集合

随后 `app.mount()` 不只是调用 renderer，还会把这个 `AppContext` 挂到根 `VNode` 上，让根组件实例在初始化时能够继承整棵应用树的全局配置。

`app.use()` 的核心不是“执行一个函数”这么简单，而是：

- 识别插件是函数还是对象形式
- 保证同一个插件不会被重复安装
- 把当前 `app` 实例作为参数传入，让插件有机会扩展全局能力

`app.component()`、`app.directive()`、`app.mixin()` 本质上都是往 `AppContext` 写入注册表，后续组件解析、指令挂载、选项合并都会读这些注册表。

`app.provide()` 则是在应用级 `provides` 上写值。根组件初始化时会把这份对象接到组件实例的 `provides` 原型链上，于是后代组件可以通过 `inject` 查到应用级依赖。

## 3. 为什么这在 mini-vue 里通常会被省略

教学实现只需要证明“组件可以渲染”，所以保留 `mount / unmount` 就够了。但完整 Vue 必须支持插件生态、全局注册和跨组件上下文，这就要求 `app` 成为一个真正的应用容器，而不是单次渲染的薄封装。

## 4. 理解重点

完整 Vue 里的 `app` 真正重要的不是 API 数量，而是它建立了“应用级上下文”。后面大量运行时能力，都是围绕这份上下文展开的。
