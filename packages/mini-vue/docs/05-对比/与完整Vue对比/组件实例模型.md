# 组件实例模型

## 1. 为什么完整 Vue 的组件实例会很复杂

完整 Vue 的组件实例不是“执行一下 `setup` 再调用 `render`”那么简单。它要同时承载：

- 组件输入
- 组件内部状态
- 模板渲染上下文
- 事件派发能力
- 插槽和属性透传
- 依赖注入
- 生命周期与副作用作用域

因此完整 Vue 会在组件挂载时构造一个结构很重的实例对象。

## 2. 完整 Vue 的实现原理

组件实例一般会围绕下面几类状态组织：

- 基础身份信息：`uid`、`type`、`parent`、`root`、`appContext`
- 输入侧状态：`props`、`attrs`、`slots`
- 输出和交互能力：`emit`、`exposed`
- 内部状态：`data`、`setupState`、`ctx`
- 渲染侧状态：`subTree`、`effect`、`update`、`render`
- 缓存和优化状态：`accessCache`、`renderCache`
- 作用域状态：`scope`、生命周期钩子数组、错误边界信息

这里最关键的是“分层”：

- `props` 是显式声明、可参与更新比对的输入
- `attrs` 是未声明但需要透传到子树的输入
- `slots` 是函数化的子内容入口
- `setupState` 是 `setup()` 返回的数据
- `ctx` / 代理对象负责把这些来源统一暴露给模板和渲染函数

完整 Vue 不会把所有字段直接摊平给模板访问，而是通过公开实例代理拦截属性读取。这个代理内部会按顺序判断：

- 是不是 `setupState`
- 是不是 `data`
- 是不是 `props`
- 是不是公开属性，如 `$attrs`、`$slots`

这样既能保持模板访问体验一致，也能在开发环境做告警与性能缓存。

## 3. 这些复杂度为什么是必要的

如果没有这套实例模型，完整 Vue 很难同时支持：

- `setup` 与 Options API 共存
- 模板代理访问
- 属性透传和插槽
- 依赖注入
- `expose`
- 生命周期和错误边界

教学版可以把实例压缩成最小可运行对象，但生产框架必须让各种来源的状态在一个统一模型里协同工作。

## 4. 理解重点

完整 Vue 组件实例的核心价值是“把不同来源、不同生命周期、不同可见性的状态装进同一套调度和渲染模型里”。
