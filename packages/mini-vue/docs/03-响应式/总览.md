## 1. 响应式模块的目标

`packages/reactivity` 的目标很纯粹：把“状态变化”转换成“依赖副作用重新执行”。

这套机制由几类角色构成：

- 数据入口：`reactive`、`ref`
- 依赖容器：`Dep`
- 副作用执行体：`ReactiveEffect`
- 依赖收集：`track`
- 依赖触发：`trigger`
- 衍生状态：`computed`
- 观察接口：`watch`

## 2. `reactive`：对象级响应式

### 2.1 基本做法

`reactive(target)` 会为对象创建 `Proxy`，当前只使用 `mutableHandlers`：

- `get`
  通过 `Reflect.get` 取值，然后 `track(target, key)`
- `set`
  通过 `Reflect.set` 设值，然后 `trigger(target, key)`

### 2.2 深层响应式怎么来的

`get` 里如果发现取出的值仍然是对象，就会调用 `toReactive(res)`，也就是继续递归包装成响应式对象。

这意味着它采用的是“按访问递归代理”而不是初始化时深遍历。

### 2.3 缓存策略

`reactiveMap: WeakMap<object, any>` 用于缓存：

- key 是原始对象
- value 是对应 proxy

### 2.4 当前 `reactive` 的明显简化

当前版本缺少很多完整 Vue 中的重要能力：

- 没有 `readonly`
- 没有 `shallowReactive`
- 没有 `shallowReadonly`
- 没有 `markRaw`
- 没有数组、Map、Set 的专门 instrumentations
- 没有对 `delete / has / ownKeys / iterate` 的精细追踪

## 3. `track` 和 `trigger`：依赖图是怎么建立的

### 3.1 数据结构

当前依赖图是：

```text
WeakMap<target, Map<key, Dep>>
Dep = Set<ReactiveEffect>
```

### 3.2 `track`

当某个响应式属性被读取时：

1. 如果没有 `activeEffect`，直接返回
2. 找到 `target -> key -> dep`
3. 把当前 `activeEffect` 加入 `dep`

### 3.3 `trigger`

当属性被写入时：

1. 找到 `target -> key -> dep`
2. 取出 dep 里的所有 effect
3. 逐个执行或交给 scheduler

### 3.4 为什么 computed 要优先触发

`triggerEffects` 不是简单按集合顺序跑，而是：

1. 先跑 `effect.computed`
2. 再跑普通 effect

这是为了避免普通 effect 在读取 computed 时拿到旧值。

## 4. `ReactiveEffect`：副作用实体

### 4.1 它是什么

`ReactiveEffect` 可以理解成“一个可被依赖系统重新执行的函数包装器”。

它至少包含：

- `fn`
- `scheduler`
- `computed`

### 4.2 `run()` 做了什么

`run()` 目前的逻辑很简单：

1. `activeEffect = this`
2. 执行 `fn()`

### 4.3 当前实现缺失的关键能力

这里有一个很大的教学版痕迹：`stop()` 是空实现。

同时也缺少：

- effect 嵌套栈
- 依赖 cleanup
- 避免重复收集
- 递归 effect 保护
- 调试钩子

## 5. `ref`：值容器式响应式

### 5.1 为什么需要 `ref`

`reactive` 只能代理对象属性，不能直接代理一个独立的原始值。

于是 `ref` 的做法是给值包一层对象：

```ts
{ value: xxx }
```

### 5.2 `RefImpl` 的两个值

内部同时维护：

- `_rawValue`
- `_value`

其中 `_value` 在非浅层模式下会被 `toReactive`。

### 5.3 当前 `ref` 的状态

已经具备：

- `isRef`
- 依赖收集和触发
- 对对象值转 reactive

但缺少：

- `shallowRef` 对外 API
- `triggerRef`
- `customRef`
- `toRef / toRefs`
- `proxyRefs`

## 6. `computed`：惰性求值与缓存

### 6.1 基本思想

`computed` 的核心是：

- 惰性
- 缓存

当前实现通过 `_dirty` 标记来完成：

- 初始为 `true`
- 首次读取 `value` 时执行 getter，并缓存结果
- 之后只要依赖没变，重复读取直接返回缓存
- 当依赖变化时，通过 scheduler 把 `_dirty` 重置为 `true`

### 6.2 当前 `computed` 的边界

仅支持 getter 形式，缺少：

- `{ get, set }` 写法
- 调试选项
- 更完整的内部优化

## 7. `watch`：当前版本只实现了窄子集

### 7.1 它支持什么

当前 `watch(source, cb, options)` 的实现，严格说只稳定支持：

- `source` 是 reactive 对象
- `immediate`
- `deep`
- 通过 `traverse` 做深度遍历收集

### 7.2 `traverse` 的作用

深度监听的关键不是“比较所有层级”，而是“访问所有层级，让 getter 全部触发一次”。

这样嵌套属性都会把当前 watcher effect 收集进去。

### 7.3 当前 `watch` 明显不完整

和完整 Vue 相比，当前缺很多：

- 不支持 `ref` 作为 source
- 不支持 getter source
- 不支持 source 数组
- 没有 `watchEffect`
- 没有 `flush: 'pre' | 'post' | 'sync'`
- 没有 `cleanup/onInvalidate`
- 没有调试回调

更关键的是，返回的停止函数虽然调用了 `effect.stop()`，但 `stop()` 本身是空实现，所以“停止监听”在当前代码里并没有真正完成。

## 8. 响应式系统和组件系统是怎么接上的

组件挂载时，`runtime-core` 会把组件 render 包进一个 `ReactiveEffect`。

于是会发生：

1. render 执行时读取响应式数据
2. `track` 把组件 effect 记录到依赖图里
3. 数据变化时 `trigger`
4. scheduler 把组件 update 丢到微任务队列
5. 队列刷新后，组件重新 render 并 patch

这说明响应式系统本身并不直接懂“DOM 更新”，它只负责驱动副作用重跑。

## 9. 这份响应式实现像 Vue 的地方

它已经有几个非常关键的 Vue 3 核心特征：

- `Proxy` 驱动对象响应式
- `ref` 作为值容器
- `WeakMap -> Map -> Set` 依赖图
- `ReactiveEffect` 作为统一副作用抽象
- `computed` 的脏标记缓存
- `watch` 通过 effect + scheduler 组合而成

## 10. 它和完整 Vue 差在哪

最大的差距不在接口个数，而在“清理与边界”：

- 没有 effect cleanup
- 没有 effect scope
- 没有完善的 stop
- 没有 collection 专用处理
- 没有 readonly/shallow 系列
- 没有调试能力
- 没有更细粒度的触发类型区分
