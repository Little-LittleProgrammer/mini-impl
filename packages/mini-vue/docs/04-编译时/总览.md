## 1. 编译层的目标

编译器解决的问题是：

> 用户写的是 `template`，运行时真正需要的是 `render` 函数。

当前 mini-vue 的编译链路是：

```text
template
-> baseParse
-> transform
-> generate
-> code string
-> new Function(code)()
-> render function
```

## 2. 编译器模块怎么分

### 2.1 `compiler-core`

负责编译主流程：

- AST 定义
- parser
- transform
- codegen

### 2.2 `compiler-dom`

当前几乎只是透传：

- `compile(template, options)` 直接调用 `baseCompile`

### 2.3 `vue-compat`

这个包把编译结果真正接到运行时里：

1. 调用 `compile(template)`
2. 得到 `{ code }`
3. `new Function(code)()` 生成 render
4. `registerRuntimeCompiler(...)`

这样组件在只有 `template`、没有 `render` 的情况下，也能在运行时完成编译。

## 3. Parse：模板怎么变 AST

### 3.1 解析器的总体思路

`parse.ts` 采用的是非常典型的“消费 source 字符串”模式：

- `context.source` 始终保存剩余未解析模板
- 每消费一段字符串，就 `advanceBy(...)`
- 递归地构建 AST 节点

### 3.2 当前能识别的核心节点

已经覆盖的基础节点包括：

- `ROOT`
- `ELEMENT`
- `TEXT`
- `INTERPOLATION`
- `ATTRIBUTE`
- `DIRECTIVE`

这意味着它已经具备模板编译的最基础表达能力：

- HTML 元素
- 纯文本
- `{{ }}`
- 普通属性
- 形如 `v-if` 的指令壳子

### 3.3 `parseChildren` 的工作方式

循环解析，直到：

- 模板耗尽
- 或遇到当前祖先节点对应的结束标签

它会按优先级识别：

1. `{{ ... }}` 插值
2. `<tag>` 元素
3. 普通文本

### 3.4 Element 和属性解析

`parseElement` 的顺序是：

1. 解析开始标签
2. 递归解析 children
3. 解析结束标签

`parseAttributes` 会在开始标签内循环处理属性，并把带 `v-`、`:`、`@`、`#` 前缀的条目归类成 `DIRECTIVE` 节点。

但要注意，这里很多高级信息还没有真正展开：

- 指令参数 `arg` 基本没做
- 修饰符 `modifiers` 基本没做
- `v-for` 没有专门 transform
- `v-on / v-bind` 没有完整处理链

## 4. Transform：AST 怎么被改写

### 4.1 transform 的职责

`transform` 阶段不是简单遍历 AST，而是把“模板语义”改写成“更接近 render code 的中间表示”。

当前 transform 上下文会维护：

- `currentNode`
- `parent`
- `childIndex`
- `helpers`
- `replaceNode`

其中最关键的是：

- 深度优先遍历
- 进入阶段收集 `onExit`
- 退出阶段倒序执行

### 4.2 `transformText`

它负责把相邻的：

- 文本节点
- 插值节点

合并成 `COMPOUND_EXPRESSION`。

例如：

```vue
<div>hello {{ msg }}</div>
```

会被合成近似：

```ts
"hello " + _toDisplayString(msg)
```

### 4.3 `transformElement`

它会把 `ELEMENT` 节点转换为 `VNodeCall`：

- `tag`
- `props`
- `children`

但当前实现很简化：

- `vnodeProps` 直接是空数组
- 没有把普通属性完整转换成对象表达式
- 没有事件、动态绑定、patchFlag 分析

### 4.4 `transformIf`

这是当前最“像真正编译器”的部分之一。

它会：

1. 把带 `v-if` 的元素替换为 `IF` 节点
2. 生成 `IF_BRANCH`
3. 在 codegen 侧转为条件表达式
4. 不满足条件时生成注释节点

但这里有一个必须指出的现实问题：

当前 `v-if` transform 会注入 `key` 之类的对象属性节点，而 codegen 对 `JS_OBJECT_EXPRESSION / JS_PROPERTY` 并没有完整生成分支，因此这条链路在“结构思路上是对的”，但在“最终代码生成完备度上还不够”。

## 5. Codegen：AST 怎么变代码字符串

### 5.1 生成上下文

`codegen.ts` 会维护一个字符串缓冲区 `code`，以及：

- `push`
- `newline`
- `indent`
- `deindent`
- `helper`

它不是在生成 JS AST 再交给其他工具输出，而是直接拼 render 函数字符串。

### 5.2 输出代码的大框架

最终生成的是类似这样的结构：

```js
const _Vue = Vue

return function render(_ctx, _cache) {
  with (_ctx) {
    const { ...helpers } = _Vue
    return ...
  }
}
```

### 5.3 已支持的节点生成

当前 codegen 已覆盖：

- `TEXT`
- `SIMPLE_EXPRESSION`
- `INTERPOLATION`
- `COMPOUND_EXPRESSION`
- `VNODE_CALL`
- `JS_CALL_EXPRESSION`
- `JS_CONDITIONAL_EXPRESSION`

### 5.4 当前 codegen 的缺口

缺口也很明显：

- 没有 `JS_OBJECT_EXPRESSION`
- 没有 `JS_PROPERTY`
- 没有完整 props 对象生成
- 没有事件处理表达式生成
- 没有 `v-for`
- 没有 slot codegen
- 没有静态提升输出
- 没有 patch flag / dynamicProps 输出

## 6. 编译器与运行时的衔接方式

### 6.1 运行时编译入口

在组件初始化时：

- 如果组件没有 `render`
- 但有 `template`
- 且 `compile` 已经通过 `registerRuntimeCompiler` 注册

那么就会在 `finishComponentSetup` 中即时编译模板。

### 6.2 这意味着什么

这意味着当前 mini-vue 支持的是“运行时编译”思路，而不是“构建时预编译”思路。

优点是：

- 理解简单
- 教学演示方便

缺点是：

- 首次渲染需要现场编译
- 没有 SFC 编译链
- 没有静态分析优化
- 没有产物层面的 tree-shaking 细分

## 7. 编译层最像 Vue 的地方

它已经完整保留了 Vue 编译器最关键的三段式：

1. `parse`
2. `transform`
3. `generate`

并且已经出现了几个真实 Vue 编译器中的核心概念：

- AST
- helper 收集
- 退出阶段 transform
- 结构性指令转换
- `VNodeCall`
- 条件表达式 codegen

## 8. 编译层和完整 Vue 的真实差距

当前最大的差距不只是“少几个指令”，而是少了几乎整个生产级编译优化体系：

- 没有 SFC 编译
- 没有 `script setup`
- 没有样式作用域编译
- 没有静态提升
- 没有 block tree
- 没有 patch flag 生成
- 没有 tree flattening
- 没有 SSR codegen
- 没有 hydration 相关输出
- 没有完整 DOM transform
