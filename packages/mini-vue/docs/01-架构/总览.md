# mini-vue 项目总架构

## 1. 目录结构怎么分层

`packages/mini-vue` 基本沿用了 Vue 3 monorepo 的拆包思路，但只保留了最核心的几个包：

```text
packages/mini-vue
├─ packages/shared
├─ packages/reactivity
├─ packages/runtime-core
├─ packages/runtime-dom
├─ packages/compiler-core
├─ packages/compiler-dom
├─ packages/vue
└─ packages/vue-compat
```

它们之间的职责关系可以概括为：

- `shared`
  提供跨包复用的基础工具，比如 `isObject`、`hasChanged`、`ShapeFlags`、`normalizeClass`。
- `reactivity`
  负责状态变更可追踪，包括 `reactive`、`ref`、`computed`、`effect`、`watch` 所依赖的底层机制。
- `runtime-core`
  负责平台无关的运行时核心，包括 `VNode`、组件实例、渲染器、调度器、生命周期。
- `runtime-dom`
  把 `runtime-core` 落到浏览器 DOM，提供 `insert / createElement / patchProp` 等宿主能力。
- `compiler-core`
  负责模板到 render code 的通用编译流程：解析、转换、代码生成。
- `compiler-dom`
  当前只是 `compiler-core` 的薄封装。
- `vue`
  聚合对外导出，相当于用户最终 import 的入口包。
- `vue-compat`
  这里并不是真正的 Vue 兼容层，而是把编译器注册到运行时，形成“带编译能力的完整版入口”。

## 2. 模块依赖关系

从源码依赖方向看，主链路大致如下：

```text
shared
  ↑
reactivity
  ↑
runtime-core
  ↑
runtime-dom
  ↑
vue

compiler-core
  ↑
compiler-dom
  ↑
vue-compat
  ↑
vue
```

这里最关键的设计点有两个。

### 2.1 运行时与平台分离

`runtime-core` 不直接操作 DOM，它只依赖一组宿主接口：

- `createElement`
- `insert`
- `remove`
- `setElementText`
- `patchProp`
- `createText`
- `setText`
- `createComment`

因此它理论上可以适配别的平台。`runtime-dom` 的作用就是把这一组抽象接口落到浏览器环境。

### 2.2 编译器与运行时通过注册打通

组件初始化时，如果组件自己没有 `render`，但有 `template`，`runtime-core/src/component.ts` 会尝试调用运行时注册进来的 `compile`：

1. `vue-compat` 从 `@vue/compiler-dom` 拿到 `compile`
2. `compileToFunction()` 把生成的代码字符串包进 `new Function(code)()`
3. `registerRuntimeCompiler(compileToFunction)` 注册到运行时
4. 组件初始化时按需把 `template` 编译成 `render`

这就是这份 mini-vue 中“编译时”和“运行时”的连接点。

## 3. 对外导出层怎么组织

最终入口是 `packages/vue/src/index.ts`：

- 导出 `@vue/reactivity`
- 导出 `@vue/runtime-core`
- 导出 `@vue/runtime-dom`
- 导出 `@vue/vue-compat`
- 导出 `@vue/shared`

这意味着当前打包产物实际上是一个“大一统入口”，既带运行时，也带编译器注册逻辑。

## 4. 当前 mini-vue 的三条核心闭环

### 4.1 闭环一：响应式驱动组件更新

```text
reactive/ref 读属性
-> track 收集 activeEffect
-> 组件 render 被 ReactiveEffect 包裹
-> 数据变化时 trigger
-> scheduler 把组件 update 放进微任务队列
-> effect.run 重新执行 render
-> renderer.patch 更新 DOM
```

### 4.2 闭环二：VNode 驱动宿主更新

```text
h/createVNode
-> 生成 vnode
-> renderer.render
-> patch 按 vnode 类型分发
-> processElement / processComponent / processText / processFragment
-> hostInsert / hostPatchProp / hostRemove
```

### 4.3 闭环三：模板变成 render 函数

```text
template
-> baseParse 生成 AST
-> transform 改写 AST / 收集 helper
-> generate 生成 render code
-> new Function(code)()
-> render 执行时返回 vnode
-> renderer.patch 渲染
```

## 5. 这份架构最像 Vue 的地方

它与真实 Vue 最接近的不是 API 数量，而是“分层方法”：

1. 用 `shared` 放公共能力
2. 用 `reactivity` 提供底层状态语义
3. 用 `runtime-core` 管平台无关逻辑
4. 用 `runtime-dom` 做宿主适配
5. 用 `compiler-core/compiler-dom` 负责模板编译
6. 在运行时与编译器之间只保留一个清晰的接缝

## 6. 这份架构明显简化的地方

虽然架构分层是对的，但很多模块只有“骨架”，并没有达到完整 Vue 的深度：

- `runtime-core` 有组件挂载和更新框架，但组件更新分支很薄，很多场景未实现
- `patchChildren` 中仍有多个 `TODO`
- `unmount` 只是简单 `hostRemove(vnode.el)`，没有递归卸载和组件级清理
- `watch` 只覆盖很窄的 source 类型
- `effect.stop()` 为空实现
- `compiler-core` 已经有 `v-if` transform，但 codegen 对对象表达式等节点支持不全

所以更准确地说，这个项目是“架构分层像 Vue，能力覆盖像教学 Demo”。
